Introduction:

        Wind turbine operators face a competitive marketplace for selling power. The historically low price of natural gas and the amazing success of solar and wind power has created thin margins. This has greatly benefited customers and the environment. Since the upfront costs are fixed, the operating cost of the wind farm can be the difference between making money and losing money in the long-term. In recent years there have been advances in technology and economies of scale that have made great improvements in the uptime of equipment. One such improvement is vibration analysis also called condition monitoring. These techniques have been widely adopted in other industries with high capital costs and strict uptime constraints (i.e. aircraft, gas turbines, etc.). Vibration analysis is a way to interpret data from accelerometers attached to various parts of rotating equipment. Changes in the vibration signals can predict equipment failure in advance. This allows for repairs to be better planned before catastrophic damage has taken place.

        Pattern Energy is one such wind turbine operator that utilizes this approach to drive down operating costs. They have graciously allowed us to sample their data for this study. Vibration sensors are installed on various rotating parts inside the wind turbine. In this study, we will focus on data from the main bearing. This bearing connects the low-speed shaft from the hub to the nacelle structure where it can engage the gearbox. The gearbox will then step up the rotation speed for proper engagement with the generator where the electricity is produced. We focus on the main bearing here due to known failures in the sample data set.

Vibration Analysis:

        The goal of vibration analysis is simple: convert raw acceleration data into actionable information but teasing apart all the sources of vibration in a complex system. Figure 2 is an illustration of how multiple sources of vibration can add together. The grey, gold, and green lines represent various sources of vibration from the bearing raceways, rolling elements, or shaft. The RED line is the sum of the three separate signals. The data collected from the sensor will only give the RED summation. Fortunately, there is a mathematical operation that can help called the Fast Fourier Transform (FFT). Figure 3 illustrates how the FFT operation allows us to separate the sources of vibration into its components. The left time axis shows the combined signal described earlier. The right frequency axis shows the distinct peaks for the separate sources of vibration. This FFT analysis can be done for a variety of frequency ranges to give more details on certain areas of interest. Using the geometry of the equipment and the rotation speed of the shaft, we can compute the expected frequency of the components of the bearing. This allows one to diagnose specific defects based on the FFT peaks. For example, since the shaft rotation speed and the number of roller bearings are known, we can calculate the expected frequency of the outer raceway defect shown in figure 4. Why MongoDB?:

        MongoDB is a document-style database that stores data in flexible, JSON-like documents, making it easy combine data of any structure. In this context, you can think of the documents as JSON BLOBs. Unlike the MySQL BLOBs, the data can queried. The main differences between MongoDB and MySQL are that the database schema can change dynamically and the database size can scale horizontally. This means if a new wind farm is added to the database with different components or new analysis techniques are developed, there is no need to force these changes into a rigid data-type constrained schema. Also, as the volume of data increases, additional clusters can be added to the network to support this growth without having to resort to partitioning or compression. MongoDB manages the 'sharding' of the data across multiple clusters to maintain reliability and availability. The other differences are shown below in figure 8. A list of companies that use MongoDB can be found here. The terminology of MongoDB is different than MySQL and the translation is shown in figure 9. Since we are storing raw data from a variety of sensors and analysis(i.e. raw time-series, FFT, etc.), we don't want to setup a rigid schema structure for each one.

MySQL to MongoDB Migration:

        The overall goal of the code in this notebook was to generalize the pulling of data from MySQL, manipulate the data, then easily upload it to MongoDB. I did this by writing a series of functions that can pass data to one-another. Below are a breakdown of the functions and what they were used for. Most of the functions work together, but they can be used separately.

  • investigate - This function simply connects to the MySQL database, runs a query, and returns the results. It is the first step in creating a pipeline to query the data.
  • decode_object – This function was used to decode the blob returned from the MySQL database. It relies on the schema of the database to access the blob.
  • xml_to_dictionary – This function takes the decoded object from the previous function and converts everything from XML to a more useable form that MongoDB can easily understand. This utilizes the xmltodict library to do more of the heavy lifting.
  • push_to_Mongo – This function pushes two data sets (meta-data and raw-data) and inserts those objects into a defined MongoDB database and collection. The datasets should be python dictionaries.
  • set_up - The goal of this function is to write many results from an sql query to mongoDB. The main disadvantage of this is that everything is inserted in the same DB and collection. This means that you have to be very aware of what you are querying. The function takes an object that has been decoded and converted to xml (db_obj in the function), a mongodb database name, and a collection name (the db and collection are created on the fly if they do not exist). The main problem with this function is that it inserts everything at once. It created a problem with memory (crashed Mongo once).
  • chunks - The goal of this function is to break apart queries to prevent memory problems. It seems like the systems can handle about 50 queries at a time before a memory problem occurs. So I will chunk this process up. It uses a generator object to more effectively handle memory.
  • run_data - This function brings it all together, you define what you want to query and insert into the database. This allows you to define the turbines and the FFTs that you want to put into the system. The function then loops through everything, queries MySQL, and uses all of the other functions to push everything to Mongo.

def push_to_Mongo(db_name, collection_name, mdata, rdata):
    """This function takes a two python dicitonaries and inserts them into
    the defined MongoDB database and the defined collection.
    Both mdata and rdata should be python dictionaries. Use xml_to_dictionary to convert
    the decoded SQL blob (XML) to a dictionary."""
    
    if type(db_name) != str:
        db_name = str(db_name)
        
    # Connect to mongo and define database name
    client = MongoClient('localhost', 27017)
    db = client[db_name]

    collection = eval("db." + str(collection_name))
    
    data = {
        "MetaData" : mdata,
        "RawData": rdata
    }
    
    result = collection.insert_one(data)
    print('One insert: {0}'.format(result.inserted_id))

# The goal of this function is to write many results from an sql query to mongoDB
# The main disadvantge of this is that everything is inserted in the same DB and collection.
# This means that you have to be very aware of what you are querying
# The function takes an object that has been decoded and converted to xml (db_obj in the function), a mongodb databsae name
# and a collection name (the db and collection are created on the fly if they do not exist).
# The main problem with this function is that it inserts everything at once. It created a problem with memory (crashed Mongo 
# once)
def set_up(db_obj, mongo_db_name, mongo_collection):
    for x in range(len(db_obj)):
        _decoded = decode_object(db_obj[x])
        db_obj[x].pop('Record')
        _meta = db_obj[x]

PyMongo Querying and Reporting:

        One of the primary goals of switching from MySQL to MongoDB was easy access to the raw data contained within the blobs found within MySQL. The document structure of MongoDB allows us to have the benefits of the meta-data along with the raw data itself. As a proof of concept, we sought to graph any FFT, for any turbine, between any two selected time periods. To achieve this goal, we utilized the pymongo library in Python. Utilizing python also granted us access to other python libraries that could prove helpful for reporting tools. Specifically, we utilized matplotlib to generate two dimensional plots of the FFT data. Ultimately, we created a function that takes a turbine name (which correlates to a database within MongoDB), a desired FFT (which correlates to a collection within the specified database), a start date, and an end date. The dates are then used to find all times in-between the start and end date in which the FFT was triggered. In this way, we can conduct ad-hoc analysis on the FFT data that we could not do before. Matplotlib also allows us to conduct 3D analysis of the FFT data. Looking at the FFTs in 3D gives us a clearer insight to how vibrations are occurring at specific frequencies within and between selected turbines.

Conclusion and Future Work:

        We have demonstrated that not every dataset fits nicely in a traditional relational database. Combine this with the wild growth of IoT and a new solution is needed to turn this unstructured data into real value. MongoDB has shown to be easy to work with, highly flexible, and extremely scalable. This makes it a great fit for a long term choice for operational historical data storage.
